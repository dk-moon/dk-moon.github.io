대회 정보 및 결과

수행 방법

데이터

  • 데이터 사전 처리
    • PDF 문서의 용지 형식이 가로 용지, 세로 용지로 혼재
    • 가로 용지의 문서는 세로의 문서 2장이 합쳐진 형태이기에, Python에서 PDF loader 라이브러리는 상단에서 하단, 좌에서 우로 문서를 텍스르 양식으로 바꾸기 때문에 올바르게 Parsing되지 않음
    • 따라서 사전에 가로 용지 문서를 반으로 나누어 세로 용지의 문서와 유사한 형식이 되도록 처리 수행
  • PDF Loader
    • 대표적인 라이브러리인 “fitz”와 “pdfplumber”라이브러리 활용
    • fitz 성능 : 최대 F1-score 0.62
    • pdfplumber 성능 : 최대 F1-score 0.69
    • PDF 사전 처리 + pdfplumber 성능 : 최대 F1-score 0.71
  • 아쉬운 점
    • PDF 문서에서의 그림과 표를 자연어 텍스트 양식으로 인식 성능이 부족
    • 수작업으로 일부 변환하는 작업을 진행을 하였지만, 실제 적용까지는 프로젝트의 기간 부족으로 인하여 적용하지 못한 점

Embedding

  • 사용 모델
    • intfloat/multilingual-e5-large
    모델기본 구조주요 학습 목적
    KoBERTBERT한국어 일반 문장 이해
    KoRoBERTaRoBERTa한국어 일반 문장 표현 정제
    multilingual-e5-largeT5 기반텍스트 간 의미 유사도 (semantic search, dense retrieval)에 특화됨
    • multilingual-e5-large 모델의 경우, text2vec 형태로 문장 임베딩 생성에 특화되어 있기에, 의미 기반 검색(semantic search), 정보 검색(retrieval), 질문-문서 매칭 같은 작업에 뛰어난 성능.
      • 재정정보 검색/분류/매칭과 같은 작업에서는 multilingual-e5 계열 모델이 더 적합하기에 KoBERT, RoBERTa 보다 성능이 높게 나옴
  • 아쉬운 점
    • GPU 자원의 한계로 인하여 더 큰 데이터로 사전 학습되어진 모델을 사용하지 못하였던 점
    • “bespin-global/klue-sroberta-base-continue-learning-by-mnr”, "Alibaba-NLP/gte-multilingual-base", "BAAI/bge-multilingual-gemma2” 등 더 다양한 임베딩 모델과 성능을 비교하지 못했던 점

VectorDB

  • 사용한 Vector Store : FAISS
  • 선정 이유
    • 쉽고 빠르게 적용이 가능한 VectorDB이며, 문서간 유사도 계산 성능이 준수
  • 아쉬운점
    • 짧은 기간 내에 결과를 도출해야 하며, 컴퓨팅 자원의 한계로 인하여 다양한 VectorDB를 활용 및 비교를 해보지 못 함

Large Language Model

  • 사용 모델 : meta-llama/Meta-Llama-3.1-8B-Instruct
  • 선정 이유
    • 다양한 오픈 소스 모델 중, 가장 답변 생성 능력이 뛰어남
    • 주어진 GPU 자원에서 수행할 수 있던 최적의 모델
  • 비교 모델
    • gemma-2 → Llama에 비하여 답변 생성 능력이 부족
  • 아쉬운 점
    • GPU 자원이 풍부하였다면, 더 큰 가중치를 가지고 있는 모델을 이용한 테스트도 수행해볼 수 있었을 것
    • LoRA를 처음 시도해보면서 끝까지 학습을 시킬 수 없었던 점
